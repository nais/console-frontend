# urql client and helpers

This directory contains the urql client and helpers used throughout the
app. It replaced the previous HoudiniGraphQL setup; see the (now
deleted) `REMOVE_HOUDINI.md` for the migration history.

## Layout

- `client.ts` — `createUrqlClient(...)`. Owns the exchange chain
  (`cacheExchange` → auth `mapExchange` → `ssrExchange` → optional
  `subscriptionExchange` → `fetchExchange`).
- `context.ts` — re-exports `setContextClient` / `getContextClient` from
  `@urql/svelte`. Use those names directly.
- `load.ts` — `runQuery(event, doc, vars)` for use inside `+page.ts` /
  `+layout.ts` `load` functions. Returns `{ data, errors }` and flips
  `isAuthenticated` on `Unauthorized` errors.
- `mutation.ts` — `runMutation(event, doc, vars)` for SvelteKit form
  actions (mirrors `runQuery`'s shape and error normalization).
- `pagination.ts` — `readCursorPagination(url, pageSize)` and
  `cursorPaginationLoaders(url, pageInfo)` for URL-driven pagination.
- `fragment.ts` — re-exports `useFragment` (and `getFragmentData` alias)
  from the generated `fragment-masking` module.
- `subscription.ts` — `subscriptionExchange` backed by `graphql-sse`.
- `gql/` — generated by `npm run codegen`. Do not edit by hand.
- `queries/` — typed query / mutation / fragment documents that are
  shared across routes. Per-route documents live next to the route
  (e.g. `src/routes/team/[team]/members/members.ts`).

## Authoring a GraphQL document

Documents are written with the generated `graphql()` helper, imported
under the alias `gql` from `$lib/urql/gql`:

```ts
import { graphql as gql } from '$lib/urql/gql';

export const ThingQuery = gql(/* GraphQL */ `
	query Thing($slug: Slug!) {
		team(slug: $slug) {
			id
			slug
		}
	}
`);
```

The `graphql as gql` alias is a project convention rather than a hard
requirement: `codegen.ts` is configured with
`pluckConfig.globalGqlIdentifierName: ['gql']`, so only `gql(...)` call
sites are extracted as urql operations. New code should follow the same
convention.

After editing or adding a document, run `npm run codegen` (or have
`npm run codegen:watch` running in the background). Generated types
land in `src/lib/urql/gql/`.

## Loading data in `+page.ts` / `+layout.ts`

```ts
import { runQuery } from '$lib/urql/load';
import { ThingQuery } from './thing';

export async function load(event) {
	return {
		Thing: await runQuery(event, ThingQuery, { slug: event.params.team })
	};
}
```

The return shape is a plain `{ data, errors }` object. In the
component, consume it directly — no Svelte store unwrap:

```svelte
<script lang="ts">
	import type { PageProps } from './$types';

	let { data }: PageProps = $props();
	let { Thing } = $derived(data);
</script>

<GraphErrors errors={Thing.errors} />
{#if Thing.data}
	…
{/if}
```

## URL-driven pagination

Use `readCursorPagination` server-side and `cursorPaginationLoaders`
client-side. See `src/routes/+page.svelte` and
`src/routes/team/[team]/members/{members.ts,+page.ts,+page.svelte}` for
worked examples.

```ts
// +page.ts
return {
	Things: await runQuery(event, ThingsQuery, {
		team: event.params.team,
		...readCursorPagination(event.url, 25)
	})
};
```

```svelte
<!-- +page.svelte -->
<script lang="ts">
	import { page } from '$app/state';
	import { cursorPaginationLoaders } from '$lib/urql/pagination';
</script>

<Pagination
	page={Things.data.team.things.pageInfo}
	loaders={cursorPaginationLoaders(page.url, Things.data.team.things.pageInfo)}
/>
```

Read filter / sort state directly from `page.url.searchParams` and
write it back with `changeParams` from `$lib/utils/searchparams` when
the UI changes.

## Mutations and refetches

Use `getContextClient()` from `@urql/svelte` (re-exported by
`$lib/urql/context`):

```svelte
<script lang="ts">
	import { invalidateAll } from '$app/navigation';
	import { getContextClient } from '$lib/urql/context';
	import { DeleteThingMutation } from './thing';

	const client = getContextClient();

	async function onDelete(id: string) {
		const result = await client.mutation(DeleteThingMutation, { id }).toPromise();
		if (result.error) {
			// surface result.error.graphQLErrors / result.error.message
			return;
		}
		await invalidateAll();
	}
</script>
```

`invalidateAll()` re-runs the `load` function, which re-issues the urql
query. Until `@urql/exchange-graphcache` is wired in, this is the
default cache-update strategy.

For SvelteKit form actions (`+page.server.ts`), use `runMutation`
instead — it mirrors `runQuery`'s shape and runs against the
per-request urql client.

## Components that re-fetch on user input

When a component genuinely needs to re-subscribe with new variables
(rather than re-load a route), use `queryStore` from `@urql/svelte`
together with `getContextClient()`. Variables can be reactive `$state`
values — the store re-issues the query whenever they change. Examples:

- `src/routes/team/[team]/Menu.svelte`
- `src/routes/Onboarding.svelte`
- `src/routes/team/[team]/cost/TeamEnvironmentApplicationsCost.svelte`
- `src/lib/domain/issues/CriticalIssues.svelte` (a "Load more"
  pattern that increments a `pageSize` `$state` — used as a stop-gap
  for what was previously `@paginate(mode: Infinite)`).

## Subscriptions

Component-level subscription pattern (see
`src/routes/team/[team]/[env]/app/[app]/logs/Logs.svelte`):

```svelte
<script lang="ts">
	import { onDestroy } from 'svelte';
	import { getContextClient } from '$lib/urql/context';
	import { ThingSubscription } from './thing';

	const client = getContextClient();

	const { unsubscribe } = client
		.subscription(ThingSubscription, {
			/* variables */
		})
		.subscribe({
			next: (result) => {
				/* handle result */
			},
			error: (err) => {
				/* handle error */
			}
		});

	onDestroy(unsubscribe);
</script>
```

To re-subscribe with new variables, tear down and re-call inside a
Svelte effect cleanup.

## Per-mutation cache updates (planned)

When `@urql/exchange-graphcache` lands, list-mutating operations
(create / delete) will need explicit `updates` resolvers. The
convention will be to colocate them with the mutation document, e.g.:

```ts
import { graphql as gql } from '$lib/urql/gql';
import type { UpdateResolver } from '@urql/exchange-graphcache';

export const CreateThingMutation = gql(/* GraphQL */ `
	mutation CreateThing($input: CreateThingInput!) {
		createThing(input: $input) {
			thing {
				id
				slug
			}
		}
	}
`);

// Registered in createUrqlClient's graphcache config.
export const createThingUpdates: UpdateResolver = (result, args, cache) => {
	// invalidate the team's `things` list so it re-fetches with the new entry
	cache.invalidate({ __typename: 'Team', slug: args.input.teamSlug }, 'things');
};
```

Until graphcache is in place, prefer `invalidateAll()` from
`$app/navigation` after a mutation.

## Optimistic updates

These will live in the client config under `optimistic` once
`@urql/exchange-graphcache` is wired in. Until then, mutations issue
through `getContextClient()` and the page is refreshed via
`invalidateAll()`.

## Scalars and codegen optionality gotchas

- `Date` and `Time` scalars are typed as `string` in the generated
  types. Anywhere a child component wants a `Date`, pass
  `new Date(value)` at the boundary or widen the prop type to
  `Date | string`.
- Nullable selections come back as `T | null | undefined`-shaped
  optional properties. Use `??` to normalize when writing to the URL
  or comparing for equality.
- Enums keep their schema `SCREAMING_SNAKE_CASE` names
  (`OrderDirection.ASC`, `AlertState.FIRING`) thanks to
  `config.namingConvention.enumValues: 'keep'` in `codegen.ts`. Use the
  enum value rather than a string literal so codegen's enum types
  accept it.

## What not to do

- **Don't** create a urql `Client` per component. Either pull it from
  context (`getContextClient()`) or, in a `load`, pass `event` to
  `runQuery` so it picks up the per-request client from
  `event.locals.urql`.
- **Don't** read `data.X.variables` — urql results don't expose the
  variables they were issued with. Read filter/pagination state from
  `page.url.searchParams` instead.
- **Don't** call `data.X.fetch(...)` — there is no client-side refetch
  on the load result. Use `invalidateAll()` from `$app/navigation`,
  or switch to `queryStore` for components that need to re-subscribe
  with new variables.
